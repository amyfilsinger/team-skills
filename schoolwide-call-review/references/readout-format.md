# Readout Format

The final deliverable is always saved to `schoolwide-call-review.md` in the
working directory. Use this exact structure.

---

```markdown
# Schoolwide Call Review

**Window:** <start date> → <end date> (<N> days)
**Calls scanned:** <total> · **Calls with schoolwide discussion:** <filtered>
**Reps covered:** <names or "team total">

---

## 1. What kinds of schoolwide conversations are happening?

Group calls into 3–5 conversation patterns. For each pattern:

### <Pattern name> (<N> calls)
- **Who's in the room:** school leader / district leader / CIO / mixed
- **Stage of buying cycle:** discovery / evaluation / pilot / renewal
- **The core "why now":** one sentence on what's driving the conversation
- **Example calls:** Call title — rep name — date

Keep patterns concrete. "School leader at partial-adoption school weighing
schoolwide upgrade" is useful. "Various discussions" is not.

---

## 2. Why school leaders are choosing to go schoolwide

Rank-order the reasons leaders are *pulling* toward schoolwide. Cite the
verbatim language they used wherever possible.

### <Reason> (mentioned in <N> calls)
> "Direct quote from a call if available."
> — Attendee title, Call title, <date>

How this maps to the enablement pillars: <which of the 5 pillars this lines up with>

---

## 3. Objections and concerns we heard

For each distinct objection, include:

### <Objection> (<N> calls)
- **How it was raised:** verbatim quote or close paraphrase
- **How the rep responded:** summary of the response
- **Enablement's recommended handling:** from the Expected Objections section
  of the playbook
- **Verdict:** strong / partial / missed — did the rep's response match the
  enablement answer?

---

## 4. What the sales team is doing well

3–6 concrete things, each tied to a specific call where it showed up. Keep this
grounded in evidence, not generic praise.

- **<Strength>:** What the rep did, why it worked, which call.

---

## 5. Where the team can improve

3–6 concrete coaching opportunities. For each:

- **<Gap>:** What was missed or mishandled
- **Which call(s):** concrete examples
- **The enablement-aligned move:** what the playbook would have had them say
- **Why it matters:** what outcome this change would likely improve

Be specific and actionable. "Talk more about outcomes" is not useful.
"Lead with the 2x engagement stat on partial-adoption calls instead of feature
descriptions" is useful.

---

## 6. Enablement alignment scorecard

| Enablement element | Calls that hit it | Calls that missed it | Notes |
|---|---|---|---|
| Discovery questions before pitch | X/N | | |
| Names ≥3 of the 5 value pillars | X/N | | |
| Cites a proof-point stat | X/N | | |
| References a proof-point account | X/N | | |
| Reframes objections per playbook | X/N | | |
| Frames schoolwide as amplifying existing adoption | X/N | | |
| Ties to a specific leader priority | X/N | | |

---

## 7. Recommended next actions

- **For the team:** 1–3 coaching points to raise at the next team meeting
- **For enablement:** 1–2 gaps in the current materials this review surfaced
  (if any — don't invent them)
- **Calls worth a deeper listen:** 2–3 calls worth pulling the whole team into
  for a listening session, with the reason for each

---

## Appendix: Call-by-call notes

For each call included in the analysis, a short block:

### <Call title> — <rep> — <date>
- **Attendees:** names + titles
- **Schoolwide discussion:** 1–3 sentence summary
- **Objections raised:** bulleted
- **Pillars named:** which of the 5
- **Stats/proof points cited:** which
- **Outcome / next step:** what the rep committed to
```

---

## Output guardrails

- **Ground every claim in evidence.** If you can't cite a specific call or
  quote, either leave the claim out or mark it "impressionistic".
- **Don't invent stats.** If a rep said "we see a lot of engagement", write
  that. Don't upgrade it to "2x engagement" unless they actually said it.
- **Don't sugarcoat misses.** The point of the review is to improve. If the
  team is consistently missing a pillar, say so plainly — with data.
- **Absolute dates, not relative.** "On 2026-04-15" not "last Tuesday".
- **Names of reps are fine** if they are on the call metadata. Attendee names
  from customer side are also fine — these are internal reviews.
